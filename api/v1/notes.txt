///////////////////// DISTRIBUTED TRANSACTION QUEUE //////////////
Because currently:- (HTTP request waits for provider)
    user request → provider request → wait provider
→ response
Better architecture:- API receives request
    → debit wallet
    → queue transaction
    → worker processes provider
    → webhook/reconciliation finalizes

***** TRANSACTION QUEUE + WORKERS (Architecture becomes:)
Client Request
→ Validate
→ Debit Wallet
→ Create Pending TX
→ Push Queue
→ Return immediately

BACKGROUND WORKER
→ Processes queue
→ Calls provider
→ Updates TX
→ Reconciliation handles unresolved

******************
SO YOUR QUEUE STATUS SHOULD REALLY BE:
    pending
    processing
    awaiting_callback
    awaiting_reconciliation
    completed
    failed
YOUR TRANSACTION STATUS SHOULD BE:
    pending
    processing
    success
    failed
    reversed
    timeout
*************************

FINAL CORRECT FLOW
    USER REQUEST
        → debit wallet
        → tx pending
        → queue pending

    WORKER
        → provider request

IF SUCCESS
    tx success
    queue completed

IF PENDING
    tx processing
    queue awaiting_reconciliation

WEBHOOK/RECON
    → tx success/failed
    → queue completed/failed

This is proper async fintech architecture.
